Active Investigation MN × SFDC — Renewal Opportunity Mapping Race Condition
MN Support Investigation

Auto-renewals missing their Salesforce opportunity

Renewal contracts are being created without a Salesforce opp attached. No opp means no contact role, broken ARR attribution, and anything downstream that reads that field silently fails. Root cause isn't missing data or a config error. An early renewal script is firing for accounts that should be on auto-renew only, leaving a cancelled draft holding the opp reference hostage. By the time MN runs its own renewal logic, it sees that reference, assumes occupied, and skips mapping. Account renews. Opp never connects.

Path A · T − 90 days
90-day renewal API script
Customer-built. Gives CSMs a working draft 90 days before renewal fires. Right idea, wrong account scope.
What it does
  • 1Fires 90 days before contract renewal date
  • 2Finds which Salesforce opportunity belongs to that account's renewal
  • 3Calls MN API to create a draft renewal quote
  • 4Stamps SFDC opp ID on draft via monetize__OriginalOpportunity__c, connecting CRM deal to MN renewal object
Why it was built

CSM-managed accounts can't wait for MN to fire a renewal 24 hours out. Quotes need negotiating, pricing sign-off, customer approval — that takes weeks. Script creates a pre-built draft, already tied to an SFDC record, so CSMs have something to work with the moment they start the renewal conversation.

Where it goes wrong
Assumes a human picks up that draft. For CSM-managed accounts, someone does. For auto-renewing accounts, nobody ever will. MN still fires its own renewal on schedule regardless, and now two processes are claiming the same SFDC opp with no awareness of each other.

Account targeting is what breaks here. MN behavior on both sides is working exactly as built.

Path B · T − 90 days
Same script, wrong accounts
Runs for every account approaching renewal, not just CSM-managed ones. Path A and Path B stop being separate right here.
What's actually happening

Script fires for all accounts approaching renewal. For accounts set to auto-renew in MN, no human will ever touch that draft. But it still gets created, still stamps an SFDC opp, and sits there until MN's expiry automation eventually cancels it.

At that point there's a cancelled quote holding an opp reference that was never released. Field is still populated. Opp still looks taken.

Why it matters
Two independent renewal processes are now touching the same account. Customer's API script ran at T−90. MN auto-renewal fires on renewal date. Neither knows about the other. Both are looking at the same SFDC opp — only one wins and it won't be the right one.

Creating the draft isn't what causes this. It's what that draft leaves behind once cancelled.

Why both sides think they're right

Both are right in their own lane. Problem is they share a lane for auto-renewing accounts and nobody accounted for it.

Root cause
Opp availability check is too shallow
MN checks whether a quote reference exists on an opp. Doesn't check if that quote is still alive. One missing condition behind all of this.
Current logic (simplified)
// Current check
IF opportunity.referenced_by_quote EXISTS
  → treat as occupied → skip mapping

Guard was built to stop one SFDC opp being attached to two active renewal quotes at once. Reasonable. But it treats a cancelled quote the same as an active one — reference exists, MN walks away, mapping never happens.

What it should do
// Required logic
IF opportunity.referenced_by_quote EXISTS
  AND quote.status IN (draft, confirmed, active)
  → treat as occupied

IF quote.status IN (expired, cancelled, voided)
  → treat opp as available

Why it took this long to surface

Missing opps were caught and corrected by hand for a long time — low volume, quiet problem. As auto-renewing contract volume scaled up, manual patching couldn't keep pace and the pattern became impossible to ignore.

Worth flagging: MN already does precedence handling for amendments — confirmed amendment kills competing drafts. That same thinking needs to reach the opp-search layer for renewals.
Short-term fix · Process
Scope API script to managed accounts only
Stops new occurrences from today. Both renewal paths run independently, on the right accounts, with no overlap.
Steps
  • 1Define a flag in SFDC or MN marking accounts as CSM-managed — likely already exists as account type or CSM assignment field
  • 2Add a pre-flight check to API script: IF account.renewal_type != 'csm_managed' → skip
  • 3Unmanaged accounts stay on MN auto-renewal only — no early draft, no occupied opp reference, no collision
Net result: CSM accounts get their 90-day head start. Auto-renewing accounts go straight to MN. Neither path touches the other's opp.
What this won't fix

Historical renewals already sitting with monetize__OriginalOpportunity__c = null. Pull those contracts, identify correct SFDC opp for each, correct manually or run a backfill script to re-link them.

Long-term fix · System
Deep opp lookup — check quote state, not just existence
Permanent fix at logic level. Doesn't matter how many external triggers run — MN will correctly evaluate whether an opp is genuinely in use.
Engineering change
// Find all quotes referencing this opp
quotes = SELECT * FROM quotes WHERE original_opportunity = opp.id
active = quotes.filter(q → q.status NOT IN ['cancelled', 'expired', 'voided'])

IF active.length == 0 → opp is available
ELSE → opp is occupied
